IMPLEMENTATION SUMMARY: everestctl auth login
=============================================
Branch: release-2.0 + PR #2361 merged
Spec reference: specs/005-authentication-system-redesign.md 


BACKGROUND
----------
PR #2361 added the full server-side authentication infrastructure:
  - POST /v1/auth/token  (password grant + refresh_token grant)
  - POST /v1/auth/revoke
  - internal/tokenregistry/ package (opaque refresh tokens stored hashed in k8s Secret)
  - internal/server/auth.go (handler implementation)

Before this work, the CLI had zero HTTP API authentication — every command
talked directly to Kubernetes via kubeconfig. This implementation adds the
CLI side of the login flow as described in spec 005.


FILES CREATED
-------------

1. pkg/cli/config/config.go
   - New package managing ~/.config/everest/config.yaml (mode 0600)
   - Multi-context format modelled on kubeconfig, matches spec 005 schema exactly:

       apiVersion: v1
       kind: Config
       currentContext: admin@localhost:8080
       contexts:
         - name: admin@localhost:8080
           context:
             server: http://localhost:8080
             accessToken: eyJ...
             refreshToken: everest_rt_...
             expiresAt: 2026-06-14T15:30:00Z

   - Types: Config, NamedContext, Context (all with yaml struct tags)
   - DefaultPath() — resolves os.UserHomeDir() + .config/everest/config.yaml
   - Load(path) — returns empty Config (not error) if file does not exist
   - Config.Save(path) — creates parent dir with 0o700, writes file with 0o600
   - Config.UpsertContext(name, ctx) — inserts if new, overwrites if name exists
   - Uses gopkg.in/yaml.v3 (consistent with rest of codebase)

   - TestLoad_FileAbsent — returns empty Config, no error when file missing

3. pkg/cli/auth/login.go
   - Package auth, business logic only (no cobra dependency)
   - Types: Config{Pretty bool}, LoginOptions{Server, Username, Password, ContextName}
   - Login struct with zap.SugaredLogger; logger silenced in Pretty mode (same pattern
     as pkg/accounts/cli/accounts.go)
   - NewLogin(cfg Config, l *zap.SugaredLogger) *Login
   - Login.Run(ctx, opts LoginOptions, cfgPath string) error:
       1. validateServerURL(opts.Server) — rejects non-http/https schemes (see security)
       2. normalizeServerURL(opts.Server) — strips trailing slash, appends /v1 if absent
          REASON: generated client resolves ./auth/token relative to base URL, so base
          must include /v1 for the full path to be /v1/auth/token. The /v1 in the
          operation path comes from the base URL only — it is NOT doubled.
       3. client.NewClient(normalizedURL) — uses generated client from client/ package
       4. c.CreateAuthToken with grant_type=password, username, password,
          refresh_token_delivery=body (explicitly requests token in response body)
       5. On non-200: reads body, returns descriptive error with status code
       6. Decodes AuthTokenResponse — RefreshToken is plain string (not *string) per
          x-go-type-skip-optional-pointer in schema
       7. Computes expiresAt = time.Now().Add(ExpiresIn seconds)
       8. Derives context name: username@host:port (see context name section below)
       9. Loads config from cfgPath, upserts context, sets CurrentContext, saves
      10. Prints output.Success in pretty mode
   - validateServerURL(server string) error — rejects file://, ftp://, javascript:,
     empty string, and anything not starting with http:// or https://
   - normalizeServerURL(server string) string — idempotent, handles trailing slashes
     and already-present /v1
   - defaultContextName(username, server string) string — produces username@host:port

4. pkg/cli/auth/refresh.go
   - Login.Refresh(ctx, cfgPath string) error — token rotation helper
   - Designed for use by future data-plane commands (not used by login itself)
   - Flow:
       1. Load config from cfgPath
       2. Find the current context by CurrentContext name
       3. validateServerURL on stored server (guards against tampered config)
       4. client.NewClient with normalised server URL
       5. CreateAuthToken with grant_type=refresh_token, refresh_token from config
          (RefreshTokenDelivery not set — server defaults to body delivery)
       6. On non-200: return error, config left unchanged
       7. On success: update AccessToken, RefreshToken, ExpiresAt in-place, Save
   - cfgPath is explicit (not resolved internally) for testability

CONTEXT NAME DESIGN
-------------------
Default format: username@host:port  (e.g. admin@localhost:8080)
Rationale (per mentor feedback): allows multiple accounts on the same server
to coexist as separate named contexts.

Optional override: --context-name flag lets users specify any name they want.

Username rename concern: investigated and confirmed NOT a real concern — the
accounts.Interface has no Rename/UpdateUsername method, the API has no such
endpoint, and the UI has no such feature. Usernames are immutable in the current
system. The only way to "rename" is delete + recreate, which invalidates all
tokens anyway.


SECURITY REVIEW
---------------
YAML injection: yaml.v3 escapes all hostile strings (newlines, colons, null
  bytes). Verified empirically — round-trip safe for all payloads tested.

URL scheme injection: validateServerURL() explicitly rejects file://, ftp://,
  javascript://, scheme-relative (//), and empty strings. Both Run and Refresh
  call this before making any network request.

JSON injection: not possible — response decoded into typed struct via
  encoding/json, no string concatenation.

Command injection: not applicable — no exec.Command or shell calls anywhere
  in the new code.

Path traversal: not possible — config file path is os.UserHomeDir() plus a
  hardcoded constant suffix. No user input touches the file path.


KEY TECHNICAL DECISIONS
-----------------------
1. cfgPath passed explicitly to Run and Refresh (not resolved internally) —
   makes tests use t.TempDir() without any mocking or monkey-patching.

2. /v1 appended in normalizeServerURL, not hardcoded in the client call —
   because oapi-codegen strips the server prefix from each operation path.
   The generated operation path is ./auth/token (not ./v1/auth/token), so
   the /v1 must come from the base URL passed to NewClient.

3. Both access token AND refresh token stored — access token (15min TTL) for
   immediate API requests; refresh token (30 day TTL) for Refresh() calls.
   ExpiresAt stores access token expiry so callers know when to refresh.

4. Refresh() is implemented now (not deferred) — because every future
   data-plane command will need it. One place, not duplicated per command.

5. No transparent auto-refresh wrapper yet — that belongs in a future
   pkg/cli/api/ package per spec. Refresh() is the primitive; the wrapper
   is the next layer.

6. No token storage in the cobra layer — cobra commands only handle flags,
   TUI prompts, and error printing. All token/config logic lives in
   pkg/cli/auth/ and pkg/cli/config/.
